# The hub API

The hub serves one HTTP API + SSE surface to every client — the CLI's fleet verbs, the runners' outbound connections, the [web app](./web-app.md) (D-048), and later the chat bot (see [index.md](./index.md)). The routes below are the surface as currently known, assembled from the settled decisions and the [runner contracts](../runner/contracts.md); the models these routes accept and return are in the [domain model](../domain/index.md). **Maturity: draft** — the operations are backed by decisions, but the route shapes are proposals; only `POST /questions/{id}/answer` appears verbatim in the corpus so far. The push path is store-and-forward with per-runner sequence idempotency (D-069), and the whole surface ships authless behind Tailscale first (D-018).

| Route | Description |
|-------|-------------|
| `POST /runners` | Register a runner — runner id + workspace id into the fleet registry (D-019); makes it visible on the board. |
| `GET /runners` | Fleet registry — the registered runners and their liveness, derived from `last_seen_at` against the staleness threshold (D-070). |
| `POST /runners/{id}/heartbeats` | The runner-level liveness heartbeat (D-070) — refreshes `last_seen_at`; much slower than the machine-local worker heartbeat, which never leaves the box. Returns `204`. |
| `PATCH /runners/{id}` | Set a runner's operational state — `paused` now, routing knobs post-MVP (D-043); declarative state the runner reads on its outbound pull and adheres to, never a push into the dev box (D-012). |
| `POST /routes` | Claim work — acquisition is the birth of a **complete** Route fact (D-021/D-024/D-080): the runner peeks the ready queue (`GET /chunks`), acquires the chunk's environments, and posts the route — chunk, runner, workspace, env ids; the hub accepts exactly one claim per chunk (`409` = race lost, the runner releases its bindings and moves on), and the response carries the first node envelope. |
| `POST /chunks` | Ingest work by id (D-047): the caller submits one or more PM pointers — `{source, ref}` (D-105) — specific items always, batch fine; each mints a chunk (`ch_<ulid>`) holding that pointer, pinned to the configured default graph (D-081), in the `not_ready` resting state (D-103) — visible on the board but not yet claimable. A pointer naming a source with no matching `[[pm_source]]` fails the whole request (D-107). A pointer already held by a live chunk is rejected `409` with the existing chunk id — re-ingest is legal once every prior holder is terminal (D-093). `blizzard hub ingest` fronts this route (D-074); the agent-native ingest flow is a documented pattern, not shipped machinery. |
| `POST /chunks/{id}/promote` | Promote a `not_ready` chunk to `ready` so a runner may claim it on a subsequent tick (D-103): appends the `chunk.promoted` fact and re-derives status. Idempotent — promoting an already-ready or running chunk is a no-op; `404` only when the chunk is unknown. `blizzard hub promote` and the board's not-ready card front this route. |
| `DELETE /chunks/{id}` | Discard a chunk — legal only while unacquired (D-047), and real deletion: nothing to retain, the PM item is the durable referent, and re-ingesting the same pointer mints a fresh chunk. Anything with history is stopped instead (D-067/D-074). |
| `POST /chunks/{id}/stop` | Stop a chunk — terminal operator abandonment, legal at any point after acquisition (D-067): artifacts and history retained, every later state-advancing write rejected regardless of epoch; the holding runner learns over its pull, kills the worker, and releases the environments. The acquired-chunk counterpart of discard. |
| `GET /chunks/{id}/pm-items` | Pass-through read of the chunk's PM items (D-047): the hub resolves each pointer's `source` to its configured binding and calls the vendor with that source's own credentials, returning each item's **body and comment thread** in vendor-native format with the pointer and `fetched_at` (D-074/D-106); attachments and linked items wait. Workers reach this read through their runner's proxy route, never directly (D-084). |
| `PATCH /chunks/{id}` | Queue-shaping metadata (D-048): set a ready chunk's queue position — the ordering mechanism is the queue-ordering open question; facts append, order derives (D-004). |
| `POST /chunks/{id}/group` | Group chunks (D-048): merge the named unacquired chunks into this one — the combined chunk carries the union of their PM pointers (D-047); the merged-away chunks are discarded. Route shape is an open question. |
| `GET /chunks` | Fleet-wide chunk status — derived by query from recorded facts, never a stored status column (D-004). |
| `GET /chunks/{id}` | One chunk in full — the chunk is the aggregate root: current node, transition history, route (runner/workspace/env — D-021), questions, and its artifact store inline (`{node}.{artifact-name}.{epoch}` entries, latest-by-epoch resolution — D-036). Payloads are modest, so the whole aggregate travels together; this is what feeds later nodes' envelopes and the board's chunk view. |
| `POST /chunks/{id}/transitions` | Submit a node-step's completion: judgement choice + check results + artifacts, one atomic epoch-fenced write (D-027, D-036) — `git_commit` branches already pushed to the forge (D-026); stale epochs rejected (D-007). This is the advancement checkpoint (D-072): the hub applies the edge and any pending migration atomically, and the **response carries the chunk's next node envelope** (or parked-at-gate / hub-node-taken), which is what lets the runner continue in place. At a human-judged node the hub **rejects** a transition — human signoff required; the completion goes to `/decisions` (D-045) — and the resolving transition arrives here referencing its `decision_id`. |
| `POST /chunks/{id}/decisions` | Record a node-step's completion as an open **Decision** (D-045): the choice set plus the step's artifacts, one atomic epoch-fenced write — the gate's counterpart of a transition; the chunk derives `waiting_on_human`. A runner may submit a decision in place of a transition for *any* node: that is the runner-config gate (D-032). |
| `POST /decisions/{id}/resolution` | Resolve a decision — a person picks one choice (the board's buttons, D-042); first-write-wins CAS like an answer. The holding runner picks the resolution up over its pull and records the resolving transition (D-027 intact). |
| `POST /questions` | Record an ask — the runner forwards a worker's question; the durable question row is born here ([ask-answer.md](../ask-answer.md)). |
| `GET /questions` | Open questions — answered-ness derived from the presence of an answer row; the board's and `blizzard hub status`'s question list. |
| `POST /questions/{id}/answer` | Answer a question — first-write-wins CAS; the loser is told who already answered ([ask-answer.md](../ask-answer.md)). |
| `GET /graphs` | The workflow graphs — standalone immutable definitions with their names, pointers, and enabled state (D-025/D-040); the authoring schema is [graph-schema.md](./graph-schema.md) (D-071). |
| `POST /graphs` | Mint a graph — validates per D-071 (errors reject, warnings flag), inlines every file reference, mints immutably (D-033/D-040), enabled by default (D-039); there is no in-place update route. |
| `PATCH /graphs/{id}` | Update a graph's operational metadata — `enabled`, the auto-migration-target gate (D-039); `auto_migrate` (`deferred`/`off`, D-037); and `migration_target`, the standing drift intent (D-040). The definition itself is immutable (D-033), so the metadata is the only patchable surface; each patch appends the facts the flags derive from (D-004). |
| `POST /chunks/{id}/migrations` | Create a migration **request** — an intent row in the chunk's collection, distinct from the applied migration record (D-037); `mode: deferred` (default — applied by the hub at the chunk's next transition ingestion, atomically with it, name-matched node — D-072) or `mode: forced` — revokes the lease via a new epoch and redoes the current node: the hotfix path (D-034). |
| `GET /chunks/{id}/envelope` | The chunk's current node envelope, idempotent (D-090) — the recovery read when a transition's apply-response is lost: a flush acked as already-applied (seq ≤ high-water mark) points the runner here. |
| `POST /chunks/{id}/detach` | Forcibly release the chunk from its runner (D-088): appends `route.released`, the chunk re-derives `ready`, and the next claim's lease floor (D-035/D-044) fences the old runner's late writes; the detached runner learns over its pull — kill the worker, release the environments. The superadmin's "this runner is gone." |
| `POST /chunks/{id}/check-delivery` | Immediately check a parked chunk's external PR state against the delivery binding (D-065) — the on-demand form of the hub's slow poll; any terminal PR state (merged, or closed without merging) completes the chunk as delivered. Idempotent, so a future forge webhook simply triggers this check. |
| `GET /merge-queue` | The deliver node's merge queue — what is queued, landing, landed (D-030); strict FIFO, one chunk at a time across all its repos (D-057), conflicts routed to the conventional `merge` graph (D-058). |
| `POST /events` | Batched event push from runners — the runner-minted facts that travel (`lease.minted`, `escalation.recorded`, `answer.delivered` — [events.md](../domain/events.md), D-067); store-and-forward always, idempotent by per-runner monotonic seq against the hub's high-water mark (D-069). |
| `GET /events/stream` | The SSE stream — hub-landed facts re-broadcast under their canonical names ([events.md](../domain/events.md), D-067); runners subscribe outbound, the board live-updates from it. |

What is deliberately **not** here: code and transcripts (D-012 — git-commit artifacts are the only reference to code), machine-local execution facts (leases, heartbeats, pids, env bindings — the [runner store](../runner/store.md)'s territory; lease facts and their epochs *do* land here via `POST /events`, because the transition fence consumes them — D-044), and any route that reaches into a worktree — a resume command only works on the machine holding the environment, which is what keeps the remote surface structurally unable to touch code.
