# The hub API

The hub serves one HTTP API + SSE surface to every client — the CLI's fleet verbs, the runners' outbound connections, the [web app](./web-app.md) (D-048), and later the chat bot (see [index.md](./index.md)). The routes below are the surface as currently known, assembled from the settled decisions and the [runner contracts](../runner/contracts.md); the models these routes accept and return are in the [domain model](../domain/index.md). **Maturity: draft** — the operations are backed by decisions, but the route shapes are proposals; only `POST /questions/{id}/answer` appears verbatim in the corpus so far. The wire-level details (batching, dedup, idempotent ingestion keys) are an [open question](../../decisions/open-questions.md), and the whole surface ships authless behind Tailscale first (D-018).

| Route | Description |
|-------|-------------|
| `POST /runners` | Register a runner — runner id + workspace id into the fleet registry (D-019); makes it visible on the board. |
| `GET /runners` | Fleet registry — the registered runners and their liveness (liveness signal itself is an open question). |
| `PATCH /runners/{id}` | Set a runner's operational state — `paused` now, routing knobs post-MVP (D-043); declarative state the runner reads on its outbound pull and adheres to, never a push into the dev box (D-012). |
| `POST /routes` | Acquire work — acquisition *is* the birth of Route facts (D-021/D-024): the runner posts its capacity, the hub selects ready chunks (its queue, its prioritization), creates the routes, and returns them with their node envelopes (prompt, node config, relevant artifacts). |
| `POST /chunks` | Ingest work by id (D-047): the caller submits one or more PM pointers — (provider, base URL, remote id) — specific items always, batch fine; each mints a chunk holding that pointer. Whether a `blizzard ingest` verb fronts this route, and whether a canned agent-native ingest flow ships, is an open question. |
| `DELETE /chunks/{id}` | Discard a chunk — legal only while unacquired (D-047): chunks are ephemeral, the PM item is the durable referent, and re-ingesting the same pointer mints a fresh chunk. Discard mechanics (who may discard; what happens to artifacts and history) are an open question. |
| `GET /chunks/{id}/pm-items` | Pass-through read of the chunk's PM items (D-047): the hub composes the real API call from each pointer and its own per-vendor credentials, returning the body in whatever format the vendor provides. Route shape and exposed fields beyond the body are an open question. |
| `PATCH /chunks/{id}` | Queue-shaping metadata (D-048): set a ready chunk's queue position — the ordering mechanism is the queue-ordering open question; facts append, order derives (D-004). |
| `POST /chunks/{id}/group` | Group chunks (D-048): merge the named unacquired chunks into this one — the combined chunk carries the union of their PM pointers (D-047); the merged-away chunks are discarded. Route shape is an open question. |
| `GET /chunks` | Fleet-wide chunk status — derived by query from recorded facts, never a stored status column (D-004). |
| `GET /chunks/{id}` | One chunk in full — the chunk is the aggregate root: current node, transition history, route (runner/workspace/env — D-021), questions, and its artifact store inline (`{node}.{artifact-name}.{epoch}` entries, latest-by-epoch resolution — D-036). Payloads are modest, so the whole aggregate travels together; this is what feeds later nodes' envelopes and the board's chunk view. |
| `POST /chunks/{id}/transitions` | Record a node-step's completion: judgement + node transition + its artifacts, one atomic epoch-fenced write (D-027, D-036) — `git_commit` branches already pushed to the forge (D-026); stale epochs rejected (D-007). At a human-judged node the hub **rejects** a transition — human signoff required; the completion goes to `/decisions` (D-045) — and the resolving transition arrives here referencing its `decision_id`. |
| `POST /chunks/{id}/decisions` | Record a node-step's completion as an open **Decision** (D-045): the choice set plus the step's artifacts, one atomic epoch-fenced write — the gate's counterpart of a transition; the chunk derives `waiting_on_human`. A runner may submit a decision in place of a transition for *any* node: that is the runner-config gate (D-032). |
| `POST /decisions/{id}/resolution` | Resolve a decision — a person picks one choice (the board's buttons, D-042); first-write-wins CAS like an answer. The holding runner picks the resolution up over its pull and records the resolving transition (D-027 intact). |
| `POST /questions` | Record an ask — the runner forwards a worker's question; the durable question row is born here ([ask-answer.md](../ask-answer.md)). |
| `GET /questions` | Open questions — answered-ness derived from the presence of an answer row; the board's and `blizzard status`'s question list. |
| `POST /questions/{id}/answer` | Answer a question — first-write-wins CAS; the loser is told who already answered ([ask-answer.md](../ask-answer.md)). |
| `GET /graphs` | The workflow graphs — standalone immutable definitions with their names, pointers, and enabled state (D-025/D-040); schema is an open question. |
| `POST /graphs` | Mint a graph — every edit mints a new immutable graph (D-033/D-040), enabled by default (D-039); there is no in-place update route. |
| `PATCH /graphs/{id}` | Update a graph's operational metadata — `enabled`, the auto-migration-target gate (D-039); `auto_migrate` (`deferred`/`off`, D-037); and `migration_target`, the standing drift intent (D-040). The definition itself is immutable (D-033), so the metadata is the only patchable surface; each patch appends the facts the flags derive from (D-004). |
| `POST /chunks/{id}/migrations` | Create a migration **request** — an intent row in the chunk's collection, distinct from the applied migration record (D-037); `mode: deferred` (default — re-pins at the next transition, name-matched node) or `mode: forced` — revokes the lease via a new epoch and redoes the current node: the hotfix path (D-034). |
| `GET /merge-queue` | The deliver node's merge queue — what is queued, landing, landed (D-030); queue mechanics are an open question. |
| `POST /events` | Batched event push from runners — leases, routes, verdicts; buffered through hub outages and flushed with idempotent ingestion keys (open question). |
| `GET /events/stream` | The SSE stream — `question.answered` and the rest of the event vocabulary (open question); runners subscribe outbound, the board live-updates from it. |

What is deliberately **not** here: code and transcripts (D-012 — git-commit artifacts are the only reference to code), machine-local execution facts (leases, heartbeats, pids, env bindings — the [runner store](../runner/store.md)'s territory; lease facts and their epochs *do* land here via `POST /events`, because the transition fence consumes them — D-044), and any route that reaches into a worktree — a resume command only works on the machine holding the environment, which is what keeps the remote surface structurally unable to touch code.
